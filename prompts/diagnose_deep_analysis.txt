Provide a deep analysis of the task, focusing on likely failure modes and root causes.

Return JSON only. Do not wrap the response in markdown or code fences.

Required schema:
{
  "task_id": 123,
  "fault_type": "comprehension|modeling|implementation|edge_handling|surface",
  "sub_tags": ["short_snake_case_tag"],
  "severity": "low|medium|high",
  "description": "one concise diagnosis summary",
  "evidence": {
    "wrong_line": "the problematic completion snippet",
    "correct_approach": "the correct reasoning or repair direction",
    "failed_test": "the failing test or error symptom"
  },
  "enriched_labels": {
    "key": "string value"
  },
  "confidence": 0.0,
  "diagnosis_source": "model_deep|non_failure",
  "prompt_version": "v1"
}

Rules:
- Use only the allowed enum values exactly as written.
- `fault_type` must be one of: comprehension, modeling, implementation, edge_handling, surface.
- `severity` must be one of: low, medium, high.
- `diagnosis_source` must be `model_deep` for real failures and `non_failure` for success-like records.
- `evidence` must be an object, never a string.
- All `enriched_labels` values must be strings.
- Use canonical weakness tags in `sub_tags`; do not invent synonyms when one of these applies.
- `function_name_mismatch` means the public entry point or test-harness contract uses a different callable name/signature than the completion exposes.
- `markdown_formatting` means the completion adds markdown fences or other raw-output formatting noise around otherwise executable code.
- `missing_code_block` means the completion gives explanation, commentary, or analysis instead of executable code.
- If the record is success-like rather than a real failure, use only success-like `sub_tags` such as `correct_output`, `correct_execution`, `correct_solution`, or `pass`, and set `diagnosis_source` to `non_failure` instead of any failure source.
- If you are unsure, still choose the closest allowed enum value instead of inventing a new one.
